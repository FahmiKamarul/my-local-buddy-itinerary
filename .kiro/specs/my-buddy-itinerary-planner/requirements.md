# Requirements Document

## Introduction

MY Buddy is a mobile-first, hyperlocal AI itinerary planner for Malaysia. It guides users through a Tinder-style card-swiping session to capture their preferences and interests, then uses those signals to generate a personalised day-trip itinerary with three distinct route options. The app accounts for real-world Malaysian conditions — traffic, queues, parking, and "Malaysian timing" — through a mandatory Human Error Buffer applied to every activity. The tone is warm and distinctly Malaysian, using local slang throughout.

---

## Glossary

- **App**: The MY Buddy Next.js web application.
- **User**: A person planning a day trip in Malaysia.
- **Destination**: A Malaysian state or city entered by the User (e.g. Melaka, Penang, Kuala Lumpur).
- **Card**: A single unit of content displayed during the swipe session — either a Clarifying Question Card or an Activity Card.
- **Clarifying Question Card**: A Card that asks the User a preference question (e.g. "Do you like street food?").
- **Activity Card**: A Card representing a specific attraction or activity, containing Title, Location, Base Duration, Price, and Priority.
- **Swipe Session**: The interactive phase where the User swipes through a deck of Cards.
- **Swipe Right**: A User gesture indicating interest or a "Yes" answer.
- **Swipe Left**: A User gesture indicating disinterest or a "No" answer.
- **Accepted Cards**: All Cards the User swiped right on during the Swipe Session.
- **Base Duration**: The estimated time in minutes to complete an activity, excluding buffers.
- **Human Error Buffer**: A mandatory time addition applied on top of Base Duration to account for parking, queues, and Malaysian timing. Default multiplier is 1.25 (25%); Santai Route uses 1.30 (30%).
- **Buffered Duration**: Base Duration × buffer multiplier, rounded up to the nearest 5 minutes.
- **Priority**: A ranking assigned to each Activity Card — High, Medium, or Low — used for card-dropping decisions when time is constrained.
- **Itinerary**: A time-ordered schedule of activities generated from Accepted Cards.
- **Optimized Route**: An Itinerary variant ordered to minimise total travel distance.
- **Makan-Focused Route**: An Itinerary variant that schedules food-related activities at peak meal times (breakfast 07:00–09:00, lunch 12:00–14:00, dinner 18:00–21:00).
- **Santai Route**: An Itinerary variant that applies a 30% buffer and inserts 15-minute rest intervals between activities for a relaxed pace.
- **Card Deck**: The full set of Cards generated by the AI for a given Destination and session.
- **Itinerary Engine**: The backend logic (implemented as a Vercel AI SDK Tool) that calculates Buffered Durations, enforces time constraints, and produces the three route variants.
- **calculate_itinerary Tool**: The Vercel AI SDK Tool Calling function that runs the Itinerary Engine.
- **Recognised Location List**: The App's internal list of valid Malaysian states and major cities used to validate Destination input.

## Requirements

### Requirement 1: Destination Input

**User Story:** As a User, I want to enter my destination in Malaysia, so that the App can generate relevant activity suggestions for that location.

#### Acceptance Criteria

1. THE App SHALL display a destination input field on the initial screen upon app launch.
2. THE App SHALL accept a Destination as valid if and only if it matches an entry in the Recognised Location List and contains between 1 and 100 non-whitespace characters.
3. IF the User submits an empty destination field or a whitespace-only destination, THEN THE App SHALL display an inline validation message below the input field prompting the User to enter a destination, and SHALL retain the current (empty) field value without clearing it.
4. IF the User submits a destination that does not match the Recognised Location List, THEN THE App SHALL display an inline error message below the input field in Malaysian-English tone, and SHALL retain the submitted value in the input field so the User can correct it without re-typing.
5. WHEN a valid Destination is submitted, THE App SHALL transition to the Swipe Session screen within 3 seconds.

---

### Requirement 2: AI Card Deck Generation

**User Story:** As a User, I want the AI to generate a personalised deck of cards for my destination, so that I can quickly express my preferences without filling in a long form.

#### Acceptance Criteria

1. WHEN a valid Destination is submitted, THE App SHALL invoke the AI to generate a Card Deck containing between 8 and 15 Cards.
2. THE Card Deck SHALL contain at least 3 Clarifying Question Cards and at least 4 Activity Cards.
3. THE App SHALL generate each Activity Card with the following fields: Title (string), Location (string), Base Duration (positive integer, minutes), Price (string, e.g. "Free", "RM5–RM15"), and Priority (one of: High, Medium, Low).
4. THE App SHALL validate each generated Card against the Zod Card schema before presenting it to the User.
5. IF the AI returns a Card that fails Zod schema validation, THEN THE App SHALL discard that Card and request a replacement from the AI, up to a maximum of 3 replacement attempts per invalid Card.
6. IF after 3 replacement attempts the Card Deck contains fewer than the minimum required Clarifying Question Cards or Activity Cards, THEN THE App SHALL display a user-facing error message and offer the User a button to retry deck generation from the beginning.
7. THE App SHALL generate all Card content using Malaysian-English tone, incorporating at least one Malaysian slang term (e.g. "Lepak", "Ngam", "On-the-way", "Makan", "Santai") per card description.
8. WHILE the Card Deck is being generated, THE App SHALL display a loading indicator to the User.
9. IF the AI fails to produce any valid Cards after 3 generation attempts, THEN THE App SHALL remove the loading indicator and display a user-facing error message with a retry button.

---

### Requirement 3: Tinder-Style Swipe Session

**User Story:** As a User, I want to swipe through cards one at a time, so that I can quickly indicate my preferences in an engaging, mobile-friendly way.

#### Acceptance Criteria

1. WHEN the Card Deck is generated and contains at least 1 Card, THE App SHALL display Cards to the User one at a time in a vertical, mobile-first layout.
2. WHEN the User performs a rightward swipe gesture on a Card, THE App SHALL record that Card as an Accepted Card and advance to the next Card.
3. WHEN the User performs a leftward swipe gesture on a Card, THE App SHALL discard that Card and advance to the next Card.
4. WHEN the User performs a touch-based swipe gesture on a Card, THE App SHALL register the swipe direction and respond within 100ms.
5. WHEN the User clicks the right-side button control on a Card, THE App SHALL treat the action as a rightward swipe; WHEN the User clicks the left-side button control, THE App SHALL treat the action as a leftward swipe.
6. WHEN the User swipes right on a Clarifying Question Card, THE App SHALL record the answer as "Yes" and associate it with the question text.
7. WHEN the User swipes left on a Clarifying Question Card, THE App SHALL record the answer as "No" and associate it with the question text.
8. WHILE the User is performing a swipe gesture on a Card, THE App SHALL display a directional overlay (green for right, red for left) on the Card, and SHALL remove the overlay within 300ms of the gesture completing.
9. WHEN the User swipes the final Card in the Card Deck, THE App SHALL end the Swipe Session and begin transitioning to itinerary generation within 500ms.
10. IF the User has swiped right on fewer than 2 Cards when the deck is exhausted, THEN THE App SHALL display a message encouraging the User to try again, and SHALL offer a button that resets all swipe decisions and restarts the Swipe Session from the first Card.

---

### Requirement 4: Itinerary Generation

**User Story:** As a User, I want the AI to build a realistic itinerary from my accepted cards, so that I get a practical day-trip plan that accounts for real-world delays.

#### Acceptance Criteria

1. WHEN the Swipe Session ends with 2 or more Accepted Cards, THE App SHALL invoke the calculate_itinerary Tool to generate the Itinerary.
2. THE calculate_itinerary Tool SHALL apply a Human Error Buffer multiplier of 1.25 to the Base Duration of every Activity Card to produce the Buffered Duration, rounded up to the nearest 5 minutes.
3. THE calculate_itinerary Tool SHALL use the Vercel AI SDK `maxSteps` feature with a maximum of 10 steps to iteratively recalculate the Itinerary until all Buffered Durations fit within the User's available time window.
4. IF the total Buffered Duration of all Accepted Activity Cards exceeds the User's available time window, THEN THE calculate_itinerary Tool SHALL drop Low-Priority Activity Cards first, then Medium-Priority Activity Cards, until the Itinerary fits; within the same priority tier, the card with the longest Buffered Duration SHALL be dropped first.
5. THE calculate_itinerary Tool SHALL never drop a High-Priority Activity Card to resolve a time constraint.
6. WHEN the calculate_itinerary Tool produces a valid Itinerary, THE App SHALL return it as a structured JSON object validated against the Zod Itinerary schema.
7. WHEN the Itinerary is ready, THE App SHALL display it to the User within 5 seconds of the Swipe Session ending.
8. IF the calculate_itinerary Tool cannot produce a valid Itinerary after exhausting all Low and Medium Priority cards, THEN THE App SHALL return a partial Itinerary containing only the High-Priority Activity Cards whose cumulative Buffered Duration fits within the time window, and SHALL notify the User that some activities were removed due to time constraints.
9. IF the cumulative Buffered Duration of High-Priority Activity Cards alone exceeds the User's available time window, THEN THE App SHALL notify the User that no itinerary can be generated within the given time window and SHALL prompt the User to extend their departure time or reduce the number of High-Priority cards.

---

### Requirement 5: Three Route Variants

**User Story:** As a User, I want to see three different route options for my itinerary, so that I can choose the style that suits my mood.

#### Acceptance Criteria

1. THE App SHALL present exactly three Itinerary route variants: Optimized Route, Makan-Focused Route, and Santai Route.
2. THE Optimized Route SHALL order activities to minimise total road travel distance between locations; IF two or more orderings result in equal total distance, THEN THE Optimized Route SHALL break the tie by placing the higher-Priority Activity Card earlier in the schedule.
3. THE Makan-Focused Route SHALL schedule Activity Cards with a Food category during peak meal time windows — breakfast between 07:00 and 09:00, lunch between 12:00 and 14:00, and dinner between 18:00 and 21:00; IF no Food-category Activity Card is available to fill a given meal window, THEN THE Makan-Focused Route SHALL fill that window with the highest-Priority non-food Activity Card that fits within the window's time bounds.
4. THE Santai Route SHALL apply a Human Error Buffer of 30% to the Base Duration of all activities and insert a minimum 15-minute rest interval between every pair of consecutive activities.
5. IF the total scheduled duration of the Santai Route — including all 30% Human Error Buffers and 15-minute rest intervals — exceeds the User's available time window, THEN THE Santai Route SHALL drop the lowest-Priority Activity Cards one at a time until the schedule fits within the time window.
6. THE App SHALL display all three route variants simultaneously on the Itinerary screen, allowing the User to scroll between them.
7. WHEN the User selects a route variant, THE App SHALL highlight that variant and display the full time-ordered activity schedule with start times, Buffered Durations, locations, and prices.

---

### Requirement 6: Time Window Input

**User Story:** As a User, I want to specify my arrival and departure times, so that the itinerary fits within my actual available time.

#### Acceptance Criteria

1. THE App SHALL display arrival time and departure time input fields in HH:MM (24-hour) format, and SHALL require both fields to contain valid times before itinerary generation can proceed.
2. IF the User has not interacted with the time fields, THEN THE App SHALL pre-populate the arrival time with 09:00 and the departure time with 18:00.
3. IF the User submits an arrival time that is later than or equal to the departure time, THEN THE App SHALL display a validation error message below the time fields specifying that the departure time must be at least 30 minutes after the arrival time, and SHALL prevent itinerary generation until valid times are entered.
4. IF the difference between the User's departure time and arrival time is less than 30 minutes, THEN THE App SHALL display a validation error and prevent itinerary generation.
5. WHEN the User has entered a valid arrival time and a valid departure time with a gap of at least 30 minutes, THE calculate_itinerary Tool SHALL use those times as the start and end boundaries of the Itinerary time window.

---

### Requirement 7: Hyperlocal Tone and Language

**User Story:** As a User, I want the app to feel like advice from a local Malaysian friend, so that the experience is warm, relatable, and culturally authentic.

#### Acceptance Criteria

1. THE App SHALL use Malaysian-English tone — characterised by Manglish sentence structure, local honorifics (e.g. "lah", "leh", "lor"), and a conversational register — throughout all UI copy, card content, AI-generated text, loading messages, empty states, and error messages.
2. THE App SHALL incorporate at least one Malaysian slang term (from the set: "Lepak", "Ngam", "On-the-way", "Makan", "Santai", "Shiok", "Boleh") per card description, per itinerary summary, and per loading message.
3. THE App SHALL include at least one culturally specific reference per Activity Card description, drawn from: peak lunch-hour traffic (12:00–14:00), hawker stall opening hours (07:00–22:00), or weekend crowd patterns (Saturday–Sunday peak 10:00–20:00).
4. IF the AI generates card or itinerary text that contains none of the required slang terms, no Manglish particles, and no cultural context references, THEN THE App SHALL re-prompt the AI to revise the text, up to a maximum of 3 re-prompt attempts.
5. IF after 3 re-prompt attempts the AI-generated text still contains no Malaysian cultural references, THEN THE App SHALL display the text as-is and log a warning, without blocking the User's flow.

---

### Requirement 8: Mobile-First UI

**User Story:** As a User accessing the app on my phone, I want a smooth, touch-friendly interface, so that I can plan my trip comfortably on a small screen.

#### Acceptance Criteria

1. THE App SHALL render all screens in a mobile-first, vertical-scroll layout optimised for viewport widths from 320px to 428px.
2. THE App SHALL display Card content in a single-column card layout with a minimum touch target size of 44×44 pixels for all interactive elements.
3. THE App SHALL function correctly on the two most recent major versions of iOS Safari and Android Chrome.
4. WHEN the User rotates the device to landscape orientation, THE App SHALL reflow the layout so that all content is accessible without requiring horizontal scrolling and all interactive elements remain reachable.
5. IF the reflow mechanism fails to prevent horizontal scrolling in landscape orientation, THEN THE App SHALL allow horizontal scrolling as a fallback rather than clipping content.
6. THE App SHALL achieve a Lighthouse mobile performance score of 70 or above on the Itinerary screen.

---

### Requirement 9: Structured Output and Schema Validation

**User Story:** As a developer, I want all AI outputs to be validated against Zod schemas, so that the app never crashes due to malformed AI responses.

#### Acceptance Criteria

1. THE App SHALL define a Zod schema for the Card type with fields: `title` (string, 1–100 chars), `location` (string, 1–100 chars), `baseDuration` (positive integer, 1–720 minutes), `price` (string matching "Free" or "RM X" pattern), `priority` (enum: "High" | "Medium" | "Low"), and `type` (enum: "activity" | "question").
2. THE App SHALL define a Zod schema for the Itinerary type that includes: `route` (enum: "optimized" | "makan-focused" | "santai"), `activities` (array of 1–20 scheduled activity objects each containing `cardTitle` (string), `startTime` (HH:MM), `endTime` (HH:MM), and `bufferedDuration` (positive integer, minutes)), and `totalDuration` (positive integer, 1–1440 minutes).
3. WHEN the AI returns a response from a call expected to produce Card or Itinerary data, THE App SHALL parse it through the corresponding Zod schema before using the data.
4. IF a Zod parse fails, THEN THE App SHALL log the validation error and retry the AI request after a 2-second delay, up to a maximum of 2 additional retry attempts.
5. IF all retry attempts are exhausted and the Zod parse still fails, THEN THE App SHALL display a user-facing error message without logging an additional error entry for the final failure.
6. THE App SHALL use Vercel AI SDK Structured Output to enforce JSON response format on all AI calls that return Card or Itinerary data.

---

### Requirement 10: Card Round-Trip Integrity

**User Story:** As a developer, I want card data to remain consistent from generation through to itinerary output, so that the itinerary accurately reflects what the User swiped on.

#### Acceptance Criteria

1. THE App SHALL preserve the `title`, `location`, `baseDuration`, `price`, and `priority` fields of every Accepted Card with identical values from the point of Swipe Right through to the Itinerary output, without modification, truncation, or type coercion.
2. THE App SHALL pass the complete set of Accepted Cards — including all five original field values — as input to the calculate_itinerary Tool without omitting or transforming any field.
3. THE App SHALL ensure that for any Accepted Card, serialising the Card object to JSON and deserialising it back produces an object whose `title`, `location`, `baseDuration`, `price`, and `priority` fields are strictly equal to those of the original Card object.
4. IF a round-trip serialisation check fails for any Accepted Card, THEN THE App SHALL log the failure with the affected card's `title` field and SHALL not pass that Card to the calculate_itinerary Tool.
